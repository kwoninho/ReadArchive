# ReadArchive - Product Requirements Document (PRD)

> **문서 버전**: v1.0
> **작성일**: 2026-03-01
> **제품명**: ReadArchive
> **제품 유형**: 개인용 독서 관리 웹서비스
> **대상 사용자**: 독서 습관을 체계적으로 관리하고 싶은 개인 사용자

## 1. 제품 개요

ReadArchive는 개인 독서 활동을 체계적으로 관리할 수 있는 웹서비스입니다. 사용자가 책 이름만 입력하면 LLM API를 활용하여 책 정보를 자동으로 수집하고, 독서 상태를 칸반 보드 형태로 시각적으로 관리할 수 있습니다.

### 핵심 가치 제안

| 가치 | 설명 |
|------|------|
| **간편한 등록** | 책 이름만 입력하면 저자, 출판사, 요약, 표지 등 정보가 자동 수집 |
| **직관적 관리** | 3가지 상태(읽고 싶은 책 / 읽는 중 / 다 읽은 책)로 독서 현황을 한눈에 파악 |
| **개인 기록** | 메모, 별점, 독서 기간 등 개인화된 독서 기록 관리 |

## 2. User Stories & Scenarios

### 2.1 유저 스토리 목록

| ID | 우선순위 | 유저 스토리 |
|----|---------|------------|
| US-01 | **P0** | As a 독서가, I want to 책 이름을 검색하여 책 정보를 자동으로 가져오고 싶다, so that 일일이 정보를 직접 입력하지 않아도 된다. |
| US-02 | **P0** | As a 독서가, I want to 내 책 목록을 "읽고 싶은 책", "읽는 중", "다 읽은 책"으로 분류하고 싶다, so that 현재 독서 현황을 한눈에 파악할 수 있다. |
| US-03 | **P0** | As a 독서가, I want to 책의 상태를 드래그앤드롭 또는 버튼으로 변경하고 싶다, so that 독서 진행 상황을 쉽게 업데이트할 수 있다. |
| US-04 | **P1** | As a 독서가, I want to 각 책에 개인 메모를 작성하고 싶다, so that 읽으면서 떠오른 생각이나 인상 깊은 내용을 기록할 수 있다. |
| US-05 | **P1** | As a 독서가, I want to 다 읽은 책에 별점(1~5)을 매기고 싶다, so that 나만의 평가를 남길 수 있다. |
| US-06 | **P1** | As a 독서가, I want to 독서 시작일과 완료일을 기록하고 싶다, so that 독서 기간과 패턴을 추적할 수 있다. |
| US-07 | **P1** | As a 독서가, I want to 등록한 책의 상세 정보(저자, 출판사, 요약, 표지)를 확인하고 싶다, so that 책에 대한 기본 정보를 빠르게 참고할 수 있다. |
| US-08 | **P1** | As a 독서가, I want to 책 목록에서 제목이나 저자로 검색/필터링하고 싶다, so that 많은 책 중에서 원하는 책을 빠르게 찾을 수 있다. |
| US-09 | **P2** | As a 독서가, I want to 모바일 환경에서도 편리하게 사용하고 싶다, so that 이동 중에도 독서 기록을 관리할 수 있다. |
| US-10 | **P2** | As a 독서가, I want to 독서 통계(올해 읽은 책 수, 월별 독서량 등)를 확인하고 싶다, so that 독서 습관을 되돌아볼 수 있다. |

### 2.2 주요 사용자 시나리오

**시나리오 1: 신규 책 등록**
1. 사용자가 "읽고 싶은 책" 칼럼의 "책 추가" 버튼을 클릭한다.
2. 검색 모달이 열리고, 책 이름을 입력한다. (예: "클린 코드")
3. LLM API가 책 정보를 검색하여 후보 목록을 표시한다.
4. 사용자가 원하는 책을 선택하면, 자동으로 "읽고 싶은 책" 칼럼에 추가된다.

**시나리오 2: 독서 상태 변경**
1. 사용자가 "읽고 싶은 책"에 있는 책 카드를 "읽는 중" 칼럼으로 드래그한다.
2. 상태 변경과 동시에 독서 시작일이 자동으로 오늘 날짜로 기록된다.
3. 독서를 완료하면 "다 읽은 책"으로 이동시킨다.
4. 완료일이 자동 기록되고, 별점 입력 프롬프트가 표시된다.

**시나리오 3: 독서 기록 작성**
1. 사용자가 "읽는 중" 칼럼의 책 카드를 클릭한다.
2. 책 상세 페이지에서 메모를 작성한다.
3. 메모는 자동 저장되며, 작성 일시가 함께 기록된다.

## 3. Functional Requirements

### 3.1 책 검색 기능 (FR-01)

#### LLM API 활용 방식

```
[사용자 입력] → [LLM API 호출] → [구조화된 책 정보 반환] → [UI 표시]
```

- **주 API**: OpenAI GPT-4o-mini (비용 효율 최적)
- **프롬프트 전략**: 책 이름을 기반으로 구조화된 JSON 형식의 책 정보를 요청
- **반환 데이터**: 제목, 저자, 출판사, 출판년도, 페이지 수, 요약(2~3문장), 장르/카테고리
- **표지 이미지**: LLM 응답에 포함된 ISBN으로 Open Library Covers API 활용

#### 프롬프트 예시

```
다음 책에 대한 정보를 JSON 형식으로 제공해주세요.
검색어: "{사용자 입력}"

응답 형식:
{
  "candidates": [
    {
      "title": "정확한 책 제목",
      "author": "저자명",
      "publisher": "출판사",
      "publishedYear": 2024,
      "isbn": "ISBN-13",
      "pageCount": 300,
      "summary": "2~3문장 요약",
      "category": "카테고리"
    }
  ]
}

후보가 여러 개일 경우 최대 5개까지 반환해주세요.
```

#### 폴백 전략

| 단계 | 방법 | 조건 |
|------|------|------|
| 1차 | OpenAI GPT-4o-mini | 기본 검색 |
| 2차 | Google Books API (무료) | LLM API 실패 시 |
| 3차 | 사용자 수동 입력 | 모든 API 실패 시 |

- LLM API 호출 실패 시 자동으로 Google Books API로 전환
- Google Books API도 실패할 경우 사용자가 직접 정보를 입력할 수 있는 폼 제공
- 모든 검색 결과는 DB에 캐싱하여 동일 검색어에 대한 중복 API 호출 방지

### 3.2 독서 상태 관리 기능 (FR-02)

#### 상태 정의

| 상태 | 코드 | 설명 | 자동 액션 |
|------|------|------|-----------|
| 읽고 싶은 책 | `WANT_TO_READ` | 등록만 된 상태 | 등록일 자동 기록 |
| 읽는 중 | `READING` | 현재 독서 중 | 독서 시작일 자동 기록 |
| 다 읽은 책 | `FINISHED` | 독서 완료 | 완료일 자동 기록, 별점 입력 유도 |

#### 상태 전이 규칙

```
WANT_TO_READ  ──→  READING  ──→  FINISHED
      ↑               │              │
      └───────────────←┘              │
      └──────────────────────────────←┘
```

- 모든 상태 간 자유롭게 이동 가능 (역방향 포함)
- 역방향 이동 시 기존 날짜 기록은 유지하되, 새로운 이동 시 날짜 갱신
- 상태 변경 이력은 별도로 기록하지 않음 (MVP 범위)

#### 상태 변경 인터랙션

- **데스크톱**: 칸반 보드에서 드래그앤드롭
- **모바일**: 책 카드 내 상태 변경 버튼 (드롭다운 또는 버튼 그룹)
- **공통**: 책 상세 화면에서 상태 선택 드롭다운

### 3.3 데이터 모델 설계

#### 주요 엔티티

```
┌─────────────────────────────────┐
│            books                │
├─────────────────────────────────┤
│ id            UUID    PK        │
│ user_id       UUID    FK → auth │
│ title         TEXT    NOT NULL   │
│ author        TEXT               │
│ publisher     TEXT               │
│ published_year INTEGER           │
│ isbn          TEXT               │
│ page_count    INTEGER            │
│ summary       TEXT               │
│ category      TEXT               │
│ cover_url     TEXT               │
│ status        TEXT    NOT NULL   │
│ rating        INTEGER (1~5)     │
│ started_at    TIMESTAMPTZ       │
│ finished_at   TIMESTAMPTZ       │
│ created_at    TIMESTAMPTZ       │
│ updated_at    TIMESTAMPTZ       │
└─────────────────────────────────┘

┌─────────────────────────────────┐
│            notes                │
├─────────────────────────────────┤
│ id            UUID    PK        │
│ book_id       UUID    FK → books│
│ user_id       UUID    FK → auth │
│ content       TEXT    NOT NULL   │
│ created_at    TIMESTAMPTZ       │
│ updated_at    TIMESTAMPTZ       │
└─────────────────────────────────┘

┌─────────────────────────────────┐
│       search_cache              │
├─────────────────────────────────┤
│ id            UUID    PK        │
│ query         TEXT    NOT NULL   │
│ result        JSONB   NOT NULL  │
│ source        TEXT    (llm/api) │
│ created_at    TIMESTAMPTZ       │
│ expires_at    TIMESTAMPTZ       │
└─────────────────────────────────┘
```

#### 엔티티 관계

```
auth.users (Supabase Auth)
    │
    ├── 1:N ──→ books
    │              │
    │              └── 1:N ──→ notes
    │
    └── (search_cache는 사용자 무관, 공용 캐시)
```

#### RLS(Row Level Security) 정책

- `books`: 본인 소유의 레코드만 CRUD 가능
- `notes`: 본인 소유의 레코드만 CRUD 가능
- `search_cache`: 모든 인증된 사용자가 읽기 가능, 쓰기는 서버 측에서만

## 4. Non-Functional Requirements

### 4.1 기술 스택

| 계층 | 기술 | 선정 사유 |
|------|------|----------|
| **프론트엔드** | Next.js 14+ (App Router) | SSR/SSG 지원, React 생태계, 우수한 DX |
| **스타일링** | Tailwind CSS + shadcn/ui | 빠른 UI 개발, 일관된 디자인 시스템 |
| **드래그앤드롭** | @dnd-kit/core | 경량, 접근성 지원, React 친화적 |
| **상태 관리** | React Server Components + Zustand (클라이언트) | 서버 컴포넌트 활용 극대화 |
| **백엔드/DB** | Supabase (PostgreSQL) | 인증, DB, 스토리지 통합 제공, 무료 티어 |
| **인증** | Supabase Auth | 소셜 로그인(Google, GitHub) 간편 연동 |
| **LLM API** | OpenAI GPT-4o-mini | 가격 대비 성능 최적 ($0.15/1M input tokens) |
| **배포** | Vercel | Next.js 최적 배포 환경, 무료 티어 |
| **패키지 매니저** | pnpm | 빠른 설치, 디스크 효율적 |
| **로컬 개발 환경** | Docker + Docker Compose | 일관된 개발 환경 제공, 로컬 전용 (프로덕션은 Vercel) |

#### 4.1.1 Docker 개발 환경

로컬 개발 환경의 일관성을 위해 Docker 컨테이너 기반 개발 환경을 제공한다. 프로덕션 배포는 기존 Vercel 서버리스를 유지한다.

**구성 파일**

| 파일 | 역할 |
|------|------|
| `Dockerfile` | 멀티스테이지 빌드 (deps → dev → builder → runner) |
| `docker-compose.yml` | 개발 서비스 정의 (포트, 볼륨, 환경 변수) |
| `.dockerignore` | Docker 빌드 컨텍스트에서 불필요한 파일 제외 |

**실행 방법**

```bash
docker compose up
```

- `http://localhost:3000`에서 개발 서버 접근
- 소스 코드 변경 시 핫 리로드 자동 적용
- 환경 변수는 `.env.local` 파일을 `env_file`로 로드

> **참고**: Docker 없이도 `pnpm dev`로 동일하게 개발 서버를 실행할 수 있다.

### 4.2 성능 요구사항

| 항목 | 목표 수치 | 측정 방법 |
|------|----------|----------|
| 초기 페이지 로드(LCP) | 2.5초 이내 | Lighthouse |
| 책 검색 응답 시간 | 3초 이내 (캐시 미스), 500ms 이내 (캐시 히트) | 서버 로그 |

| 동시 사용자 | 개인 프로젝트 기준 1~10명 | Supabase 무료 티어 범위 |

### 4.3 비용 최적화

**월 예상 비용 (개인 사용 기준)**

| 항목 | 무료 티어 | 예상 사용량 | 월 비용 |
|------|----------|------------|--------|
| Supabase | 500MB DB, 1GB 스토리지, 50,000 MAU | 충분 | $0 |
| Vercel | 100GB 대역폭, 서버리스 함수 | 충분 | $0 |
| OpenAI API | 없음 | 월 100회 검색 기준 | ~$0.05 |
| **합계** | | | **~$0.05/월** |

**LLM API 비용 관리 전략**

1. **검색 결과 캐싱**: 동일 검색어는 DB 캐시에서 반환 (캐시 만료: 30일)
2. **경량 모델 사용**: GPT-4o-mini 사용 (GPT-4 대비 약 1/100 비용)
3. **프롬프트 최적화**: 불필요한 토큰 최소화, 응답 형식 고정
4. **일일 호출 제한**: 사용자당 일 50회 검색 제한 (남용 방지)
5. **폴백 우선**: 가능한 경우 무료 API(Google Books) 우선 시도 옵션 제공

### 4.4 보안 고려사항

| 영역 | 대응 방안 |
|------|----------|
| **인증/인가** | Supabase Auth + RLS로 사용자 데이터 격리 |
| **API 키 보호** | OpenAI API 키는 서버 측(Next.js API Route/Server Action)에서만 사용 |
| **입력 검증** | 사용자 입력 sanitize, LLM 프롬프트 인젝션 방어 |
| **HTTPS** | Vercel 기본 제공 |
| **환경 변수** | `.env.local` 사용, 민감 정보 클라이언트 노출 금지 |
| **Rate Limiting** | API Route에 rate limiter 적용 (사용자당 분당 10회) |

### 5.2 핵심 인터랙션 패턴

| 인터랙션 | 동작 | 피드백 |
|---------|------|--------|
| 드래그앤드롭 | 책 카드를 다른 칼럼으로 드래그 | 드롭 가능 영역 하이라이트, 낙관적 UI 업데이트 |
| 책 추가 | 모달 내 검색 → 결과 선택 | 토스트 메시지 "책이 추가되었습니다" |
| 상태 변경 | 드롭다운 선택 또는 드래그 | 칸반 보드 즉시 반영 |
| 메모 작성 | 텍스트 입력 → 자동 저장 | 저장 완료 인디케이터 표시 |
| 별점 입력 | 별 아이콘 클릭 (1~5) | 즉시 반영, 클릭한 별까지 채워짐 |
| 책 삭제 | 삭제 버튼 → 확인 다이얼로그 | "정말 삭제하시겠습니까?" 확인 후 삭제 |
| 검색/필터 | 상단 검색바 입력 | 실시간 필터링 (디바운스 300ms) |


#### 캐싱 상세

- **캐시 키**: 검색어를 정규화(소문자, 공백 정리)하여 사용
- **캐시 만료**: 30일 (책 정보는 변경 빈도가 낮음)
- **캐시 적중률 예상**: 사용자가 겹치는 경우 40~60% 예상
- **캐시 무효화**: 수동 (관리자 개입 시에만)

### 6.3 API 엔드포인트 설계

> Next.js Server Actions를 주로 사용하되, 필요 시 API Route를 병행합니다.

#### 책 관련 API

| 메서드 | 엔드포인트 | 설명 | 인증 |
|--------|-----------|------|------|
| GET | `/api/books` | 사용자의 책 목록 조회 (상태별 필터 가능) | 필수 |
| POST | `/api/books` | 새 책 등록 | 필수 |

| PATCH | `/api/books/[id]` | 책 정보 수정 (상태, 별점, 날짜 등) | 필수 |
| DELETE | `/api/books/[id]` | 책 삭제 | 필수 |

#### 검색 API

| 메서드 | 엔드포인트 | 설명 | 인증 |

| 메서드 | 엔드포인트 | 설명 | 인증 |
